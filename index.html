Skript Addon

pathetic-skript

A* pathfinding for Skript, powered by the Pathetic engine. Calculate paths between locations on your server — asynchronously, without freezing the main thread.

v1.0.0 Skript 2.15.0+ Paper Java 17+

Overview

pathetic-skript wraps the Pathetic A* pathfinding library into Skript syntax. It exposes two ways to find a path between two locations:

  • Async (recommended) — fire-and-retrieve pattern using an effect + expression pair. The pathfinder runs off the main thread; the script resumes once the result is ready.
  • Sync (simple) — a single expression that blocks until the path is found. Convenient for short paths or testing, but can cause lag on complex routes.
✓ The async pattern is recommended for production servers. It offloads all computation and never blocks your tick loop.

Installation

Drop the pathetic-skript-1.0-SNAPSHOT-all.jar into your server's plugins/ folder. The -all jar is a fat jar and includes all required Pathetic dependencies — no separate installs needed.

⚠ Skript must be installed before this addon loads. The plugin will fail silently if Skript is absent.

EffPathfindStart

Starts an async pathfinding operation and stores the result under a string ID. The script thread waits for the result without blocking the main server thread, then continues.

start [calculating] path[finding] from %location% to %location% with id %string%
Effect Async
Description

Starts the async pathfinder. Does not block the main server thread. Use ExprAsyncPathfind with the same ID to retrieve the computed locations.

Parameters
ParameterTypeDescription
%location%LocationThe start position of the path
%location%LocationThe target position of the path
%string%StringA unique ID used to retrieve the result later
Since

1.0.0

Example
start calculating path from {_start} to {_target} with id "mypath"
ℹ This effect uses Skript's AsyncEffect API — the script pauses here on a separate thread until pathfinding completes, then resumes automatically. No wait statement is needed.

ExprAsyncPathfind

Retrieves the list of locations that form the path computed by EffPathfindStart. Always use this in combination with the async effect — call the effect first, then read the result with this expression.

calculated path %string%
Expression Async
Returns

Location (multiple) — all waypoint locations along the computed path, in order from start to target.

Parameters
ParameterTypeDescription
%string%StringThe ID passed to EffPathfindStart
Description

Get a path from a location to a location async. Does NOT block the main server thread. In most cases, prefer this over ExprPathfind.

Since

1.0.0

Example
set {_nodes::*} to calculated path "mypath"

ExprPathfind

A single synchronous expression that calculates and returns the path in one step. Easier to write, but blocks the main server thread during calculation. Best suited for short paths or offline/async contexts.

path[find] from %location% to %location%
Expression Blocking
Returns

Location (multiple) — all waypoint locations along the computed path.

Parameters
ParameterTypeDescription
%location%LocationThe start position of the path
%location%LocationThe target position of the path
Description

Get a path from a location to a location. Blocks the main server thread until the path is found.

Since

1.0.0

Example
set {_nodes::*} to path from {_start} to {_target}
# shorthand also works:
set {_nodes::*} to pathfind from {_start} to {_target}
⚠ This expression runs on the main thread and will cause lag for long or complex paths. Prefer the async effect + expression pair on production servers.

Async Pattern

The recommended workflow is a two-step pattern: first start the pathfinding with an ID, then read the result by that same ID. Because EffPathfindStart is an AsyncEffect, the script waits on a background thread — the main thread is free the entire time.

async-pattern.sk 
# 1. Start pathfinding (waits off main thread until done)
start calculating path from {_start} to {_target} with id "route-1"

# 2. You would want to wait a few ticks (5 for short paths, more for longer paths)
set {_nodes::*} to calculated path "route-1"

loop {_nodes::*}:
    # each loop-value is a Location along the path
    teleport player to loop-value
    wait 1 tick

Full Examples

Navigate a mob to a player

mob-navigate.sk 
on right click:
    set {_mob} to nearest zombie within 30 of player
    if {_mob} is set:
        set {_start} to location of {_mob}
        set {_target} to location of player

        # Use the async pattern — no lag
        start calculating path from {_start} to {_target} with id "mob-path"

        set {_nodes::*} to calculated path "mob-path"
        send "Path has %size of {_nodes::*}% waypoints." to player

Sync path (simple use)

sync-path.sk 
command /path <location>:
    trigger:
        set {_nodes::*} to path from location of player to arg-1
        if {_nodes::*} is not set:
            send "No path found!" to player
            stop
        send "Found a path with %size of {_nodes::*}% steps." to player

Unique IDs per player

When multiple players run pathfinding at the same time, give each a unique ID to avoid overwriting each other's results:

per-player-id.sk 
set {_id} to "path-%uuid of player%"
start calculating path from {_start} to {_target} with id {_id}
set {_nodes::*} to calculated path {_id}
